# Phase L: `project-guide init --no-input` Plan

> Generated by `plan_phase` mode on 2026-04-10. This plan combines a mini-concept, feature requirements, and technical changes for Phase L. Once approved, the corresponding stories will be added to `docs/specs/stories.md`.
>
> **Driver:** the upstream spec [`project-guide-no-input-spec.md`](project-guide-no-input-spec.md), which was drafted as the dependency contract for pyve's `pyve init` post-hook integration (pyve Phase G / Story G.c / FR-G2). Pyve cannot ship that integration until `project-guide init` can run unattended without hanging on stdin or exiting non-zero on a re-run.

## Mini-Concept (Why)

Today, `project-guide init` is safe to run once interactively but hostile to automation in two specific ways:

1. **Non-idempotent re-run.** Re-running `init` on an already-initialized project calls `click.Abort()` (exit 1) at [cli.py:90-96](../../project_guide/cli.py#L90-L96). For an interactive user, that is a reasonable guardrail. For a subprocess caller like `pyve init` — which may invoke `project-guide init` every time the user runs `pyve init`, including repeat runs in the same project — this makes re-runs spuriously fail and forces pyve to either parse the error, pre-check for `.project-guide.yml`, or pass `--force` (which is too aggressive).
2. **No "unattended" signalling surface.** `init` has no interactive prompts in the current codebase, so the practical impact of adding `--no-input` today is mostly signalling (documenting intent), CI ergonomics (`CI=1` auto-detect), and **future-proofing** — so that any prompt added to `init` later automatically inherits the correct unattended behavior instead of silently breaking the pyve integration a year from now.

Phase L closes both gaps with a single small-scoped change: make `init` idempotent on re-run, add a `--no-input` flag plus auto-detection for CI and non-TTY stdin, and route any current or future prompts through a shared `should_skip_input()` helper that honors all four triggers.

This phase deliberately does **not** broaden `--no-input` to other commands (`purge` has a `click.confirm()`, but it is not in the pyve integration path). If we later want `purge --no-input` or `update --no-input`, the shared helper is ready and the story-level scope will be a simple wire-through.

## Gap Analysis

| Capability | Today | Needed |

| Failure surface documentation | Not documented | README + CLI `--help` describe `--no-input` behavior and the four triggers |

## Feature Requirements (What)

### FR-L1. Idempotent `init` re-run

Re-running `project-guide init` on a project that already has a valid `.project-guide.yml` exits **0** with a brief confirmation message and performs no file I/O.

**Behavior:**
- If `.project-guide.yml` exists and `--force` is **not** given: print `project-guide already initialized at <target_dir>/ (use --force to reinitialize).` to stdout and exit 0.
- If `.project-guide.yml` exists and `--force` **is** given: current behavior (overwrite template tree, re-render `go.md`, rewrite config) unchanged.
- If `.project-guide.yml` does **not** exist: current behavior unchanged.
- The idempotency check is based solely on the presence of `.project-guide.yml`. A mixed state (config absent but target directory populated) falls through to the existing `_copy_template_tree` skip-with-warnings path — out of scope for this phase.

**Acceptance:**
- Re-running `project-guide init` twice in a row in the same project produces exit 0 both times.
- The second run does not modify any files (no `mtime` change on template files, no `go.md` re-render).
- `project-guide init --force` on an initialized project still performs the full overwrite.

### FR-L2. `--no-input` flag on `init`

Add a `--no-input` flag to `project-guide init`. No short form. `--help` text: "Do not read from stdin; use defaults where sensible. Fail loudly if any prompt has no default. (Also auto-enabled by CI=1 or non-TTY stdin.)"

### FR-L3. Trigger priority and auto-detection

The "skip stdin" state is resolved by `should_skip_input()` in priority order, first match wins:

1. `--no-input` flag passed explicitly → skip stdin
2. `PROJECT_GUIDE_NO_INPUT` env var set to any truthy value (`1`, `true`, `yes`, case-insensitive) → skip stdin
3. `CI` env var set to any truthy value → skip stdin
4. `sys.stdin.isatty() == False` → skip stdin
5. Otherwise → interactive (read stdin normally)

The helper is a module-level function (e.g., `project_guide.runtime.should_skip_input(flag: bool) -> bool`) that takes the parsed `--no-input` flag value and inspects env + TTY. It is the **only** place these triggers are checked, so future commands can adopt unattended mode by calling the same helper.

### FR-L4. Prompt-handling contract

When `should_skip_input()` returns `True`, any prompt in `init` must resolve one of three ways:

1. **Default available** → use the default, no prompt shown.
2. **Value derivable from environment** (git config, `pyproject.toml`, existing config file, CLI flag) → use the derived value.
3. **Required value with no default and no derivation source** → exit 1 with stderr message: `ERROR: <setting> is required when --no-input is active. Provide via --<flag> or PROJECT_GUIDE_<VAR>.`

Because `init` has no prompts in the current codebase, FR-L4 ships as **contract + helper + tests only**. No production prompt code exercises it today. A unit test injects a dummy prompt through the helper to verify the missing-required-setting path raises the correct error. This test is the regression guard for the day a prompt is added to `init`.

### FR-L5. Exit codes

| Code | Meaning |
|---|---|
| 0 | Initialized successfully, or already initialized (idempotent no-op) |
| 1 | Required setting missing when `--no-input` is active (with remediation hint). Currently unreachable from production code paths; exercised only by unit test. |
| 2 | Filesystem / I/O / permission error (unchanged from current `init`) |
| 3 | Config / metadata error (unchanged) |

### FR-L6. Output behavior unchanged

`--no-input` affects **stdin only**. Stdout and stderr are unchanged — same file-by-file install lines, same `✓` markers, same error formatting. A future `--quiet` flag is out of scope.

### FR-L7. Documentation

- `project-guide init --help` documents `--no-input` with the trigger list.
- README adds a short "Unattended / CI use" section showing the four trigger mechanisms.
- `docs/specs/project-guide-no-input-spec.md` status line is updated from `Proposed (2026-04-10)` to `Implemented in vX.Y.Z` when Phase L ships.
- `CHANGELOG.md` entry covers idempotency and `--no-input` in separate bullets so downstream consumers (pyve) can cite the exact semantics they depend on.

## Technical Changes (How)

### New module: `project_guide/runtime.py`

A small new module holding the unattended-mode helper. Keeping this separate from `cli.py` is deliberate: `cli.py` is already ~770 lines and the helper is reused by multiple concerns.

```python
# project_guide/runtime.py
import os
import sys


_TRUTHY = frozenset({"1", "true", "yes", "on"})


def _env_truthy(name: str) -> bool:
    val = os.environ.get(name, "").strip().lower()
    return val in _TRUTHY


def should_skip_input(flag: bool = False) -> bool:
    """Return True when the tool must not read from stdin.

    Triggers (priority order, first match wins):
      1. explicit --no-input flag
      2. PROJECT_GUIDE_NO_INPUT env var
      3. CI env var
      4. non-TTY stdin
    """
    if flag:
        return True
    if _env_truthy("PROJECT_GUIDE_NO_INPUT"):
        return True
    if _env_truthy("CI"):
        return True
    try:
        return not sys.stdin.isatty()
    except (AttributeError, ValueError):
        # sys.stdin may be None or closed in some subprocess contexts.
        return True
```

### Changes to `project_guide/cli.py`

1. Add `--no-input` click option to `init` (boolean flag, default `False`).
2. Replace the current [cli.py:90-96](../../project_guide/cli.py#L90-L96) abort-on-exists path:
   ```python
   if config_path.exists() and not force:
       click.echo(
           f"project-guide already initialized at {target_dir}/ "
           f"(use --force to reinitialize)."
       )
       return  # exit 0
   ```
3. Compute `skip_input = should_skip_input(no_input)` early in `init` and thread it to any future prompt site. No production prompt today, so this is plumbing only.
4. Add a small sentinel helper `_require_setting(name: str, cli_flag: str, env_var: str) -> NoReturn` that raises `click.ClickException` with the FR-L4 error message and exits 1. This is the landing spot for future prompt call sites.

### Test additions

New `tests/test_runtime.py`:
- `should_skip_input` with flag, env var (`PROJECT_GUIDE_NO_INPUT` variants: `1`, `true`, `YES`, empty, unset), `CI` env var, non-TTY (monkeypatch `sys.stdin`), and the `AttributeError`/`ValueError` safety path.
- Precedence order: flag beats env, env beats CI, CI beats TTY, all beat TTY-only.

Additions to `tests/test_cli.py`:
- Re-run `init` in an initialized project → exit 0, no file changes, expected stdout message.
- `init --no-input` on a fresh project → normal install, exit 0.
- `init --no-input --force` on an initialized project → overwrite, exit 0.
- `init` with `CI=1` env var set (via `monkeypatch.setenv`) → behaves the same as `--no-input`.
- `init` with stdin redirected from a closed file (non-TTY) via `CliRunner(input=...)` → behaves the same as `--no-input`.
- `init` with the injected-missing-prompt helper → exit 1 with the FR-L4 message format.

### `.metadata.yml`

No metadata changes. Phase L is a CLI-surface + runtime change, not a mode change. No new mode, no new artifact.

### README and docs

- README: new "Unattended / CI use" subsection under the existing Quick Start or Installation section, short — just the four triggers and one example each.
- MkDocs (if affected): add `--no-input` to the command reference page for `init`.
- CHANGELOG: two bullets minimum — (a) idempotent `init` re-run, (b) `--no-input` flag + auto-detection.

## Out of Scope

Deferred to future phases — explicitly **not** in Phase L:

1. **`--no-input` for other commands.** `purge` has a `click.confirm()`; `update` has modified-file prompts. Wiring them to `should_skip_input()` is trivial but not required by the pyve integration. A follow-up phase can sweep all commands once there is demand.
2. **`--quiet` / output suppression.** The spec is explicit: `--no-input` only affects stdin. A separate `--quiet` flag for reducing stdout chatter from `_copy_template_tree` is a reasonable future request (pyve's subprocess captures all of that), but not this phase.
3. **Config-file fallback for prompt values.** `init` has no prompts that need this today. When the first one appears, the pattern is: CLI flag → env var → `.project-guide.yml` key → default. FR-L4 already describes the required error path; the fallback mechanism lands with the first real prompt.
4. **Legacy broken-state detection.** The "`.project-guide.yml` absent but target directory populated" state is unusual (partial install, manual deletion) and falls through to the existing copy-with-skip-warnings path. Not addressed by FR-L1.
5. **Interactive mode inversion.** No `--interactive` flag to force interactive mode over a non-TTY default. If someone needs it, `stdin` can always be re-attached. Out of scope.

## Dependencies and Ordering

- **No upstream dependencies.** Phase L is independent of Phase M (project-essentials integration — which itself absorbs the former K.h and K.i stories as M.a and M.b).
- **Should ship before Phase M** because Phase M is larger and Phase L unblocks the pyve integration work. That said, the two phases touch disjoint files (runtime.py + cli.py vs. render.py + templates) and could ship in either order without merge pain.
- **Spike not required.** The integration boundary (`sys.stdin.isatty()`, env vars, click options) is stdlib-level and well understood. The upstream spec already serves as the design validation.

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Idempotency check is too narrow (only looks at `.project-guide.yml`) and misses a partial-install state | Acknowledged and documented as out of scope (FR-L1 rule, Out of Scope #4). Existing `_copy_template_tree` skip-with-warnings behavior remains the fallback. |
| `CI=1` auto-detect surprises a developer who has `CI=1` set locally for some reason | Escape hatch: they can unset the var, or run interactively in a fresh shell. The explicit `--no-input` flag exists for users who want deterministic behavior. Alternative would be to drop `CI=1` auto-detect entirely — rejected because it is the whole point of the "just works" experience for CI. |
| Non-TTY auto-detect breaks interactive use through tmux/screen/some terminals that report non-TTY | `sys.stdin.isatty()` is reliable on POSIX and Windows consoles in all modern versions of Python 3.11+. If this becomes a real problem, a `--interactive` override flag is a small follow-up. |
| Future prompt gets added to `init` without routing through `should_skip_input()` | The FR-L4 test that injects a dummy prompt is the regression guard. Code review + `should_skip_input` being the single documented idiom keeps new prompts compliant. |
| Downstream (pyve) assumes a specific version of `project-guide` for the new behavior | The upstream spec already documents this: pyve does **not** pin a version; older `project-guide` versions just degrade (prompt fails on closed stdin, pyve catches non-zero exit and warns with `--no-project-guide` hint). No coupling. |

## Open Questions

None. The upstream spec's "open question" about what `init` prompts for is resolved by inspection: it prompts for nothing. FR-L4 ships as a contract + helper + test, not as active prompt handling.
